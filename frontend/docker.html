Introducción

Resumen del proyecto y objetivo técnico.

Este proyecto consiste en una plataforma web de videojuego retro (Rompe Ladrillos) diseñada bajo una arquitectura de microservicios mediante contenedores. La idea central es separar responsabilidades en tres contenedores independientes (Frontend, Backend y Database) para facilitar despliegue, escalado y mantenimiento.

Arquitectura del Sistema

Descripción de las capas y cómo se comunican.

Capas

  • Capa de Presentación (Frontend): Servidor Nginx que entrega archivos estáticos (HTML/CSS/JS) y el motor de renderizado del juego (Canvas).
  • Capa de Aplicación (Backend): API REST en Node.js + Express que gestiona la lógica del juego, endpoints y la persistencia de puntuaciones.
  • Capa de Datos (Database): PostgreSQL para almacenamiento persistente de récords y datos relacionados.

Comunicación

  • Los contenedores se comunican mediante una red interna definida en Docker Compose; el backend accede a la base de datos por el nombre de servicio (ej. db).
  • El frontend realiza llamadas HTTP al backend (por ejemplo, /api/score) para guardar puntuaciones.
  • Se recomienda usar variables de entorno para credenciales y configuración de red.

Especificaciones del Frontend

Tecnologías y detalles del motor de renderizado.

  • Tecnologías: HTML5, CSS, JavaScript (Canvas API).
  • Renderizado: El núcleo del juego usa el API Canvas de HTML5 para dibujar ladrillos, pelota y paleta.
  • Dimensiones: Canvas de 700 x 700 píxeles (configurable).
  • Lógica de juego: Archivo game.js con motor de colisiones, estados (gameOver, gameWin) y manejo del DOM para botones y puntuación.
  • Interfaz: Sistema de estados con botones de reinicio y guardado de puntuación; Nginx sirve los archivos estáticos desde /usr/share/nginx/html.
Notas de implementación: Mantener separación entre la lógica de renderizado y la lógica de persistencia (llamadas al backend). Evitar llamadas sincrónicas en el bucle de render.

Especificaciones del Backend

API REST, endpoints y manejo de la base de datos.

Stack

  • Node.js (ej. node:18-alpine en Dockerfile)
  • Express para enrutamiento y manejo de peticiones HTTP
  • pg (node-postgres) para conexión con PostgreSQL

Endpoints principales

  • POST /api/score — Recibe JSON con { player, score } y realiza INSERT en la tabla scores.
  • Validaciones: comprobar conexión a la DB antes de ejecutar operaciones; sanitizar entrada.
  • Manejo de errores: implementar reintentos (retries) al arrancar para esperar a que la base de datos esté lista.

Ejemplo básico de endpoint

/* Ejemplo simplificado (Node.js + Express) */
app.post('/api/score', async (req, res) => {
  const { player, score } = req.body;
  try {
    await db.query('INSERT INTO scores (player, score, created_at) VALUES ($1,$2,NOW())', [player, score]);
    res.status(201).json({ ok: true });
  } catch (err) {
    console.error(err);
    res.status(500).json({ error: 'DB error' });
  }
});

Infraestructura y Dockerización

Cómo se dockeriza cada componente y recomendaciones.

Dockerfiles

  • Frontend: FROM nginx:alpine, copiar archivos estáticos a /usr/share/nginx/html, exponer puerto 8080 (host).
  • Backend: FROM node:18-alpine, copiar código, ejecutar npm install, exponer puerto 3000.
  • Database: usar imagen oficial postgres:15-alpine o similar; definir variables de entorno POSTGRES_USER, POSTGRES_PASSWORD, POSTGRES_DB.

Docker Compose

El archivo docker-compose.yml define servicios, red y volúmenes.

version: '3.8'
services:
  frontend:
    build: ./frontend
    image: rompe-ladrillos-frontend:latest
    ports: ['8080:80']
    depends_on: ['backend']
    networks: appnet

  backend:
    build: ./backend
    image: rompe-ladrillos-backend:latest
    ports: ['3000:3000']
    environment:
      - DATABASE_HOST=db
      - DATABASE_USER=postgres
      - DATABASE_PASSWORD=secret
      - DATABASE_NAME=rompe
    depends_on: ['db']
    networks: appnet

  db:
    image: postgres:15-alpine
    environment:
      - POSTGRES_USER=postgres
      - POSTGRES_PASSWORD=secret
      - POSTGRES_DB=rompe
    volumes:
      - pgdata:/var/lib/postgresql/data
    networks: appnet

volumes:
  pgdata:

networks:
  appnet:
    driver: bridge
Los contenedores se comunican por nombres de servicio (ej: el backend busca el host db).

Volúmenes y persistencia

  • Usar un volumen persistente (ej. pgdata) para que las puntuaciones no se pierdan al reiniciar contenedores.
  • Para el frontend y backend, montar volúmenes en desarrollo para hot-reload; en producción usar imágenes inmutables.

Guía de Despliegue (Manual de Operación)

Pasos para construir y ejecutar el sistema en local con Docker Desktop.

  1. Requisitos: Tener Docker Desktop instalado y en estado Running.
  2. Construcción y arranque: Abrir terminal en la raíz del proyecto y ejecutar: 
    docker-compose up --build -d
  3. Acceso:
    • Juego en local: http://localhost:8080
    • API: http://localhost:3000
  4. Depuración: Ver logs con: 
    docker logs -f <nombre_contenedor>
  5. Mantenimiento: Para detener y eliminar contenedores: 
    docker-compose down

Consejos operativos

  • Si el backend falla al arrancar por falta de DB, implementar reintentos en el arranque (esperar y reintentar la conexión).
  • Comprobar que el volumen pgdata tenga permisos correctos y espacio suficiente.
  • En entornos de producción, usar variables de entorno seguras y no almacenar contraseñas en texto plano.
